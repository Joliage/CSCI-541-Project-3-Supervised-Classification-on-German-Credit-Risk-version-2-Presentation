Project Overview

A sequel to Project 2

In Project 2 we tackled German Credit Risk with classical scikit-learn methods — Logistic Regression, Decision Tree, and Random Forest. Best accuracy: ~80% (Logistic Regression).

Project 3 keeps the dataset and problem identical but pivots to deep learning. The question becomes: can a fully-connected neural network — properly tuned — outperform the linear and tree-based models on this small, mildly-imbalanced tabular dataset?

Spoiler

Yes — but only after hyperparameter tuning with Keras Tuner. A vanilla MLP underperformed the baselines (~76%); a tuned MLP reached ~87%. The lesson is more about discipline than depth.

Motivation

Why move to a neural network at all?

Reason 01

Non-linear feature interactions

Hidden layers learn combinations that single linear coefficients can't represent. Credit decisions often depend on interactions — high income only matters if checking_status looks healthy.

Reason 02

Built-in regularization

Dropout acts as ensemble averaging during training. We get noise-robust models without manually tuning L1/L2 penalties on every parameter.

Reason 03

A foundation for the future

Mastering MLPs is the gateway to CNNs, RNNs, and Transformers. Project 3 is the bridge between Project 2's classical ML and everything that follows in CSCI 541.

Géron Ch. 10 frames this as the natural next step: once a problem has been understood with shallow methods, a deeper model lets us trade interpretability for representational capacity.

Why scaling matters even more for neural networks

Gradients flow through every layer's weight matrix. If one input feature is in the thousands and another in the tenths, the gradient updates become unstable and training stalls. Standardization is the single cheapest way to keep optimization well-behaved.

Architecture · Visual

The MLP, drawn

Multi-Layer Perceptron — input 48 → 64 → 32 → 1 (sigmoid)

Each circle is a neuron; each line is a learned weight. Even with just 64 + 32 + 1 = 97 active neurons, the network has 5,249 trainable parameters (almost all of them in the dense connections).

Architecture · Code

Step 6 — Define the model

06_build_model.py 
def build_model(input_shape, learning_rate=0.001):
    model = Sequential([
        Dense(64, activation='relu',
              input_shape=(input_shape,),
              name='Hidden_Layer_1'),
        Dropout(0.3),
        Dense(32, activation='relu', name='Hidden_Layer_2'),
        Dropout(0.3),
        Dense(1, activation='sigmoid', name='Output_Layer'),
    ])

    model.compile(
        optimizer=Adam(learning_rate=learning_rate),
        loss='binary_crossentropy',
        metrics=['accuracy'],
    )
    return model

# Instantiate
input_shape = X_train.shape[1]   # 48
model = build_model(input_shape)

Sequential is the simplest Keras API: a linear stack of layers. Each Dropout(0.3) randomly zeros 30% of the previous layer's outputs during training — Géron Ch. 11's standard regularization.

Architecture · Compile choices

Why Adam, why binary_crossentropy?

Optimizer: Adam (lr = 0.001)

Adam adapts the learning rate per parameter using running estimates of the first and second moments of the gradient — Géron's default recommendation for almost any problem. lr=0.001 is the standard starting point.

Loss: binary_crossentropy

For two-class problems with sigmoid output, this is the right log-likelihood loss. It penalizes confident wrong predictions much harder than uncertain ones.

Metric: accuracy

Easy to read and matches Project 2's metric. We'll supplement with precision / recall / F1 during evaluation since the classes aren't balanced.

Géron Ch. 10 recipe

"For binary classification: one neuron with sigmoid in the output layer, binary crossentropy loss, Adam optimizer." Almost identical to our setup — and that's by design.

PART 05 / 06

Tune the
hyperparameters.

Use Keras Tuner's Hyperband algorithm to search the architecture and learning rate space — a tournament-style elimination of bad configurations.

Tuning · Hyperband

Hyperband — early elimination, more candidates

Hyperband is a smart variant of random search. Instead of training every candidate to convergence, it:

  • Starts with many random configurations, each trained for a few epochs
  • Drops the worst performers early
  • Reallocates the saved budget to the survivors, training them longer
  • Repeats — a tournament where bad models are eliminated quickly

The result: we evaluate many more candidates than naive grid search, for the same wall-clock time.

Hyperband bracket — schematic
Tuning · Search space

Step 11 — Define the tunable model

13_model_builder.py 
def model_builder(hp):
    """A Keras model whose hyperparameters are drawn from `hp` at search time."""
    model = Sequential([
        Dense(
            hp.Int('units_1', min_value=32, max_value=128, step=32),
            activation='relu', input_shape=(input_shape,),
        ),
        Dropout(hp.Float('dropout_1', min_value=0.2, max_value=0.5, step=0.1)),
        Dense(
            hp.Int('units_2', min_value=16, max_value=64, step=16),
            activation='relu',
        ),
        Dropout(hp.Float('dropout_2', min_value=0.2, max_value=0.5, step=0.1)),
        Dense(1, activation='sigmoid'),
    ])
    model.compile(
        optimizer=Adam(
            learning_rate=hp.Choice('learning_rate', [0.001, 0.0001]),
        ),
        loss='binary_crossentropy',
        metrics=['accuracy'],
    )
    return model

hp.Int, hp.Float, and hp.Choice register what to tune. Hyperband will sample from these ranges and decide which combinations survive.

Tuning · Search space

REFLECTIONS

Conclusion
& caveats.

What we learned moving from Project 2's classical ML to Project 3's neural network — and what neural networks still can't do for us on this dataset.

Key Findings

What this project taught us

01 — Untuned MLPs can lose to Logistic Regression

76% MLP vs 80% LR — depth doesn't automatically mean accuracy. Small tabular datasets reward careful tuning more than reckless capacity.

02 — Width matters more than depth here

The best architecture (128 → 64) wasn't deeper than the baseline (64 → 32) — just wider. With 700 training rows, two hidden layers is the right depth; the question was how many neurons each.

03 — Hyperband pays for itself

Searching 512 configurations would be infeasible with naive grid search. Hyperband's early-elimination gave us the +11 accuracy points in minutes, not hours.

04 — Dropout 0.3 was a sweet spot

The search settled on 0.3 for both layers — exactly Géron's recommended middle. Higher dropout under-fit; lower dropout over-fit on this dataset size.

Limitations

Where this model is still weak

  • Small dataset — 1,000 rows is tight for any neural network. Most variance in tuning scores is sampling noise. Production-grade credit models train on millions of records.
  • Class imbalance remains — even at 87%, the recall on class 0 (bad credit) is the costly metric, and it improved less than overall accuracy did. We'd want SMOTE, class weights, or threshold tuning before any deployment.
  • No early stopping or checkpointing in the final run — the project spec called for both. We trained 50 epochs blindly; a properly-monitored run would have stopped where validation loss bottomed out.
  • No regularization comparison — we only tried dropout. L2 weight decay (kernel_regularizer) might have helped further; Batch Normalization could have stabilized training.
  • Tuner objective was accuracy — for credit risk, that's the wrong target. F1, recall-on-bad, or expected cost would all be more aligned with business stakes.
  • Black-box predictions — Project 2's Logistic Regression coefficients told us why; the MLP doesn't. For regulated credit decisions, SHAP/LIME would be required.
Future Work

Where we'd take this next

Early stopping

Add EarlyStopping(patience=5) and ModelCheckpoint callbacks — the project spec's "bells and whistles" we still owe.

Address class imbalance

SMOTE-resampled training data and/or class_weight={0: 2.0, 1: 1.0} in model.fit() to focus the model on the costly minority class.

Better objective

Retune the model with objective='val_f1' or a custom expected-cost metric instead of plain accuracy.

Try richer architectures

Wide-and-deep, ResNet-style skip connections, or attention layers — all overkill for 1,000 rows but instructive comparisons.

Calibration

Run Platt scaling on the test set — turn the sigmoid output into a calibrated probability that approval thresholds can act on.

Explainability

Wrap predictions with SHAP — required for any real credit deployment under regulations like ECOA or the EU AI Act.

Recap

Project 3 in one slide

What we built

A Keras Sequential MLP for binary credit risk classification — two hidden Dense layers (128 → 64), dropout 0.3 on each, Adam optimizer at lr=0.001, sigmoid output, binary crossentropy loss.

What we proved

Hyperparameter tuning is not a footnote — it's the difference between losing to Logistic Regression (76%) and beating every Project 2 model (87%). Keras Tuner's Hyperband makes that tuning practical.

What we visualized

Learning curves showed overfitting starting around epoch 15; TensorBoard panels gave us live scalar & weight-histogram monitoring; the architecture diagram made parameter counts tangible.

What's next

Wrap predictions in SHAP explanations, add early stopping, address class imbalance, and re-tune with a recall-on-bad objective — credit risk demands all four before any real deployment.

Key Takeaways

Three things worth remembering

Takeaway 01

Defaults are a starting point, not a destination

Géron's Table 10-2 gets you off the ground. Keras Tuner gets you airborne. Skipping the tuning step is leaving accuracy on the table.

Takeaway 02

Dropout is the small-dataset best friend

With only 700 training rows, every parameter is at risk of memorizing noise. Dropout 0.3 randomly halves the effective network at each step and keeps the model honest.

Takeaway 03

Visibility is non-negotiable

TensorBoard's live curves let us see overfitting as it happens. Without that visibility, we'd train silently and only learn the model is broken at evaluation time.

Closing Thought

When to reach for a neural network

Géron's gentle warning, Chapter 10

"You can do more than 95% of classification problems with classical ML methods. Neural networks shine when data is plentiful, features are perceptual (images, audio, text), or interactions are deeply non-linear. For tabular problems with under a thousand rows, expect classical methods to be competitive — and prepare to earn any improvement with disciplined tuning."

German Credit Risk sits right on that boundary. The MLP won — by 7 points and after a 512-config search. The lesson isn't "neural networks always win." The lesson is: they only win when you do the work.
